## Template structure

### Roadmap Database
- **Feature Name (Title)**: "Mobile push notifications v2" — the exact feature name engineers and stakeholders recognise
- **Quarter (Select)**: 2026 Q3 / 2026 Q4 / Backlog / Shipped — move to Shipped only when live to users, drives the Executive Dashboard view
- **Product Area (Select)**: Core App / Notifications / Payments / Analytics / Onboarding / Integrations / Admin / Mobile — matches your team structure, filters engineering sprint views
- **Effort Score (Number)**: 1-13 using Fibonacci sequence — engineering lead sets this during grooming, auto-calculates capacity warnings in Timeline view
- **Business Impact (Select)**: Critical / High / Medium / Low — stakeholder priority level, determines which features surface in the Executive Dashboard
- **Owner (Person)**: Assign to the PM responsible for delivery — drives notification automations and accountability in weekly standups
- **Target Release (Date)**: 2026-08-15 — hard deadline for stakeholder communication, auto-sorts Timeline view and flags at-risk features
- **Dependencies (Text)**: "Requires auth service v3 (due July 30)" — critical for sprint planning, gets reviewed in weekly dependency check-ins
- **Status (Select)**: Not Started / Discovery / Development / Testing / Ready to Ship / Shipped — engineering updates this after each sprint, feeds into stakeholder reports

### Sprint Planning Database
- **Sprint Name (Title)**: "2026 Sprint 15" — follows your existing numbering system
- **Start Date (Date)**: 2026-07-08 — first day of sprint, auto-populates weekly planning calendar
- **Features (Relation)**: Links to Roadmap Database features assigned to this sprint — determines total effort score and capacity warnings
- **Team Capacity (Number)**: 34 story points — total team velocity for this sprint, flags overcommitment when exceeded by linked features

## How to use it week by week

1. **Every Monday morning**: Open the Timeline view, review any features with red warning flags (overdue by 7+ days), and update Status for features that moved last week. Paste new feature requests into the Backlog section with preliminary effort scores.

2. **Tuesday sprint planning**: Filter Roadmap Database by your assigned Product Area, drag features from Quarter view into the current Sprint Planning entry. Check that total Effort Score stays under Team Capacity number — template auto-highlights overcommitment in red.

3. **Wednesday stakeholder sync**: Open Executive Dashboard view (filters by Critical and High business impact only), export as PDF, and send to leadership. This view shows only features that affect revenue or user retention metrics.

4. **Friday retrospective**: Update Status field for all features touched this week. Move any delayed features to next quarter and update Target Release dates. Review Dependencies text for any blockers affecting next sprint.

## Frequently Asked Questions

### How do I handle features that span multiple quarters?
Break the feature into smaller components in separate Roadmap Database entries, each with its own Quarter assignment and Effort Score. Link them using the Dependencies field with clear handoff requirements like "Mobile UI complete before API integration begins."

### What happens when engineering estimates change mid-sprint?
Update the Effort Score immediately and check if your Sprint Planning database shows capacity warnings. If total effort exceeds Team Capacity by 20%+, move lowest Business Impact features to the next sprint and update stakeholders through the Executive Dashboard.

### Can I track external vendor dependencies in this template?
Yes — use the Dependencies field to note external blockers like "Payment processor API (vendor ETA: Aug 15)" and create a separate Status option called "Blocked - External" so these features don't count against your team's velocity in sprint planning.
